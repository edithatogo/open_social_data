# AGENTS.md - Guidelines for AI Agent Contributions

Welcome, AI Agent! This document provides guidelines and tips for contributing to the Open Social Datasets project. Your adherence to these guidelines will help maintain code quality, consistency, and collaboration.

## 1. General Principles

*   **Understand the Goal:** Before starting a task, ensure you understand the user's request and the overall project goals related to making social datasets accessible.
*   **Plan Your Work:** Always create a clear, step-by-step plan using the `set_plan` tool. Get user approval for the plan before significant implementation.
*   **Iterative Development:** For complex tasks, consider breaking them down. Fetching samples of data before attempting full downloads is a good practice.
*   **Communicate Clearly:** Use `message_user` for updates and `request_user_input` when clarification or decisions are needed.
*   **Log Your Work:** Maintain session logs diligently. Update `TODO.md` and `CHANGELOG.md` as appropriate when tasks are completed or their status changes.

## 2. Code and Scripting Guidelines

### 2.1. Python
*   **Language:** Python is the primary language for scripting (e.g., data fetching, processing).
*   **Dependencies:** Manage Python dependencies using the `requirements.txt` file in the repository root. Ensure it's updated if new libraries are added. Use `pip install -r requirements.txt` to set up your environment.
*   **Structure:**
    *   **Shared Scripts:** Place general-purpose scripts (e.g., API fetchers for a specific source like ABS or AIHW) in the `scripts/shared/` directory.
    *   **Dataset-Specific Scripts:** Scripts for fetching or processing a particular dataset should reside within that dataset's specific directory, typically `datasets/[source]/[dataset_name]/scripts/`.
*   **Paths:** When writing scripts, be mindful of file paths. For scripts inside dataset directories, paths to data or log subfolders should ideally be relative to the script's location (e.g., `../data/`, `../logs/`) or constructed robustly using `os.path.join` and `os.path.dirname(os.path.abspath(__file__))`. Avoid hardcoding paths that are specific to your sandbox instance if they won't work generally.
*   **Error Handling:** Implement robust error handling (e.g., `try-except` blocks) for API requests, file operations, and data processing.
*   **Logging:** Use the `logging` module for script output. Log important actions, errors, and progress. For dataset-specific scripts, log files should typically go into the dataset's `logs/` subdirectory.

### 2.2. API Interaction
*   **User-Agent:** When making HTTP requests to external APIs, include a standard browser User-Agent string in the headers if you encounter access issues (e.g., 403 errors). Example: `Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/91.0.4472.124 Safari/537.36`.
*   **API Keys:** Currently, the APIs used (ABS, AIHW) do not require authentication keys for public data access. If future datasets require API keys:
    *   **DO NOT** commit API keys directly into scripts or any files in the repository.
    *   Plan for a mechanism to use environment variables or a secure configuration method. Request user input on how to handle this if it arises.
*   **Rate Limiting:** Be mindful of API rate limits. If fetching large amounts of data, implement respectful delays if necessary, although current scripts haven't required this explicitly.

## 3. Repository Structure and File Management

*   **Datasets:** Each new dataset should have its own directory under `datasets/[source_agency_acronym]/[dataset_identifier]/`.
    *   Example: `datasets/aihw/myhospitals/`
*   **Standard Subdirectories (per dataset):**
    *   `data/`: For raw and processed data files (e.g., Parquet, CSV). Consider an `archive/` subfolder for older versions.
    *   `docs/`: For dataset-specific documentation (data dictionary, accessible guide).
    *   `scripts/`: For scripts related to fetching, processing, or analyzing this specific dataset.
    *   `logs/`: For logs generated by scripts processing this dataset.
*   **Templates:** Utilize the documentation templates found in the `/templates` directory:
    *   `dataset_readme_template.md`
    *   `data_dictionary_template.md`
    *   `accessible_guide_template.md`
    *   Populate these thoroughly for each new dataset.

## 4. Documentation

*   **Dataset READMEs:** Each dataset directory must contain a `README.md` providing an overview, access details, links to its data dictionary and accessible guide, methodology, license, etc.
*   **Data Dictionaries:** Create a detailed data dictionary for each processed dataset, explaining variables, types, and meanings.
*   **Accessible Guides:** Write a plain-language guide for each dataset to help non-technical users understand its purpose and key aspects.
*   **Session Logs:** Maintain a session log for significant tasks or dataset integrations (e.g., `logs/aihw_myhospitals_session_log.md`).
*   **`TODO.md`:** Keep this updated with current and future tasks.
*   **`CHANGELOG.md`:** Document notable changes, additions, and fixes here, following the "Keep a Changelog" format.

## 5. Testing

*   **Incremental Testing:** Test scripts incrementally. For data fetching:
    1.  Test metadata endpoints first (e.g., list categories, list datasets).
    2.  Fetch a small sample of actual data (e.g., `top=10` records) to verify parsing logic before attempting full downloads.
    3.  Test pagination logic thoroughly.
*   **Verify Outputs:** After processing, check the integrity and structure of output files (e.g., Parquet files).

## 6. Committing and Submitting

*   **Branching:** Use a descriptive branch name for your work (e.g., `feature/aihw-new-measure`, `fix/abs-api-parser`).
*   **Commit Messages:** Write clear and concise commit messages. A common format is:
    ```
    Subject line (50 chars max)

    More detailed explanatory text, if necessary. Wrap it to about 72
    characters. Explain the "what" and "why" vs. "how".
    ```
*   **Submit Tool:** Use the `submit` tool when your work for a given task or plan is complete and tested.
